# GitHub Social Preview Plan

## Purpose

This file describes what a GitHub Social Preview image should communicate for `PDF TO ESX AGENT`.

The goal is not hype. The goal is to make the repository look credible, specific, and useful to both developers and insurance-industry users.

## Current Asset Situation

No suitable logo, icon, or social-preview-ready image asset was found in the repo during the discoverability pass.

That means the image should be created later rather than improvising from unrelated assets.

## What The Image Should Communicate

- this is a real Windows desktop application
- it converts insurance estimate PDFs into structured ESX/XML outputs
- it is local, inspectable, and workflow-oriented
- it serves both technical contributors and industry users

## Recommended Headline Directions

Good headline options:

- `Convert Insurance Estimate PDFs Into ESX/XML`
- `Local PDF-To-ESX Conversion For Insurance Estimates`
- `Open-Source Insurance Estimate PDF Parser And ESX Exporter`

## Recommended Subheadline Directions

Good subheadline options:

- `Windows desktop app with OCR fallback, canonical normalization, and deterministic export`
- `Built for messy real-world estimate packets, not clean demo PDFs`
- `Local-first pipeline for parsing estimate PDFs and generating structured ESX/XML artifacts`

## Visual Composition Guidance

Recommended composition:

- left side:
  product name plus one clear headline
- center:
  simple pipeline visual such as `PDF -> OCR/Parse -> Canonical Model -> ESX/XML`
- right side:
  a small, clean representation of the output artifacts such as `*.esx`, `*.esx.xml`, `*.canonical.json`

Use a restrained visual style:

- professional blue, slate, steel, or muted teal palette
- one or two accent colors at most
- simple document and XML iconography
- clean typography with strong contrast

## How To Make It Credible For Both Audiences

For developers:

- show that this is a real pipeline, not a vague “AI” box
- keep terminology concrete
- include PDF, OCR, and ESX/XML cues

For insurance-industry users:

- use plain language around insurance estimates and structured export
- avoid developer-only jargon dominating the image
- optionally use subtle property/estimate cues, but keep them secondary to the actual conversion story

## What Not To Include

- fake dashboards or fabricated success metrics
- carrier logos or vendor branding that could imply endorsement
- exaggerated claims like “universal parser” or “100% accurate”
- generic AI robot imagery
- cluttered code screenshots
- a dense wall of small text

## Suggested Layout Variants

### Option 1: Product Utility

- strong headline
- simple pipeline
- output artifact labels
- best for credibility and clarity

### Option 2: Industry + Technical Blend

- headline plus short subheadline
- subtle house/roof/property icon in the background
- pipeline and output artifacts in the foreground
- best if the maintainer wants stronger insurance-industry resonance

## Practical Production Notes

- keep the image readable at small thumbnail size
- optimize for the GitHub share card crop, not only full-resolution viewing
- export a version that still works on both light and dark GitHub surfaces
